# Дробление фаз занятости (отрезки фазы)

## Контекст: как сейчас работают фазы

### 1. Два типа фаз в интерфейсе

| Тип | Источник данных | Формат | Где отображается |
|-----|-----------------|--------|-------------------|
| **План (планируемые фазы)** | `task_positions` (БД) → `TaskPosition` | Один непрерывный блок на задачу: `startDay`, `startPart`, `duration` | `OccupancyPhaseBar` в строке задачи |
| **Факт (фактические фазы)** | Jira changelog → `StatusDuration[]` → `StatusPhaseCell[]` | Уже несколько отрезков на задачу (по статусам) | `OccupancyActualPhases` в строке факта |

Дробление нужно только для **плановых** фаз (полоски в таймлайне плана). Факт уже приходит отрезками из истории статусов.

### 2. Модель данных плановой фазы (текущая)

- **Тип** `TaskPosition` (`types/index.ts`): одна запись на задачу (на фазу dev или QA).
  - `startDay` (0..9), `startPart` (0..2), `duration` (в частях дня; 1 часть ≈ 1/3 дня).
  - Один непрерывный диапазон ячеек: `[startCell, startCell + duration)`.

- **БД** `task_positions`: одна строка на `(sprint_id, task_id)`.
  - Колонки: `start_day`, `start_part`, `duration`, `planned_*`, `is_qa`.

- **API**: GET возвращает массив позиций; POST/batch принимает массив с полями `start_day`, `start_part`, `duration` (одна позиция на задачу).

### 3. Отображение и взаимодействие

- **OccupancyPhaseBar** (`occupancy/components/OccupancyPhaseBar.tsx`):
  - Рисует одну полосу от `startCell` до `startCell + duration`.
  - Drag: перенос всей полосы (меняется `startDay`/`startPart`, `duration` не меняется).
  - Resize: ручки слева/справа — сдвиг начала или изменение длительности.
  - Контекстное меню: вызывается `onContextMenu(e, task, false)` — общее меню задачи, не отдельное для фазы.

- Сетка таймлайна: `WORKING_DAYS = 10`, `PARTS_PER_DAY = 3` → 30 ячеек на спринт. Позиция задаётся в ячейках.

### 4. Валидация (`features/sprint/utils/occupancyValidation.ts`)

Используется преобразование «одна позиция → один диапазон»:

- `positionToCellRange(position) → { startCell, endCell }`.
- Проверки:
  1. **dev vs QA**: занятость QA не должна начинаться до окончания dev по той же задаче.
  2. **Пересечение по исполнителям**: у одного assignee все позиции проверяются на пересечение диапазонов.
  3. **Исполнитель недоступен**: дни фазы не должны попадать в отпуск/техспринт.

Для отрезков нужно: либо рассматривать каждую позицию как набор отрезков и проверять все пары отрезков (в т.ч. между задачами), либо заранее мержить отрезки одной задачи в непересекающиеся диапазоны — и тогда порядок dev/QA и пересечения считаются по объединённым диапазонам. Логичнее считать **эффективную занятость по отрезкам**: только включённые отрезки участвуют в валидации и в оценке.

### 5. Оценка и «доп часть»

- В `OccupancyPhaseBar` и карточках: `estimatedDuration = getTaskPoints(task)` (sp/tp в частях дня), `extraDuration = max(0, duration - estimatedDuration)` — показывается как «+N sp/tp».
- При дроблении фазы: **эффективная длительность** = сумма длительностей включённых отрезков. Валидация и отображение «доп части» должны опираться на эту сумму.

## Идея доработки (UX)

1. Пользователь растягивает фазу от начала до конца (как сейчас).
2. Правая кнопка мыши по полосе фазы → контекстное меню.
3. Опция **«Разбить на отрезки»** (или «Разбить фазу на отрезки»).
4. Фаза отображается **квадратами** (сетка по ячейкам или по дням); клик по квадрату — вкл/выкл отрезка.
5. Сохранение — в БД уходят только включённые отрезки (или один «контейнер» с массивом отрезков).
6. После сохранения фаза отображается уже **отрезками**; пустые (выключенные) не учитываются в валидации и в расчёте оценки/доп части.

## Реализация (отдельная таблица сегментов)

Реализовано через отдельную таблицу `task_position_segments` (миграция `database/migrations/011_task_position_segments.sql`).

- **Таблица**: `task_position_segments(sprint_id, task_id, segment_index, start_day, start_part, duration)` с FK на `task_positions` и ON DELETE CASCADE.
- **Типы**: `PhaseSegment`, `TaskPosition.segments?: PhaseSegment[]`.
- **API**: GET позиций подгружает сегменты и добавляет в каждую позицию; POST/batch записывает сегменты в таблицу (удаление старых + вставка новых).
- **Валидация и отображение**: `getPositionSegmentRanges(position)` в `occupancyUtils`; валидация переведена на проверку по всем отрезкам.
- **UI**: при наличии `position.segments` рисуются несколько полос (по одной на отрезок); ПКМ по фазе → «Разбить на отрезки» → модалка с сеткой квадратов (вкл/выкл), сохранение в отрезки.

**Перед первым использованием выполнить миграцию:**
```bash
psql -U ... -d <your-db-name> -f database/migrations/011_task_position_segments.sql
```

## Варианты реализации (исторически)

### Вариант A: Поле `segments` в одной позиции (не использован)

- **Модель**: у одной задачи по-прежнему одна запись в `task_positions`, но помимо `start_day`/`start_part`/`duration` храним массив отрезков.
  - **Семантика**: если `segments` задан и не пуст — занятость = объединение этих отрезков. Поля `start_day`, `start_part`, `duration` можно трактовать как «габарит» (мин. день/часть начала и общая длина для обратной совместимости и быстрого отображения полосы-контейнера).
  - **Хранение**: в БД — колонка `segments JSONB`, например `[{ "start_day", "start_part", "duration" }, ...]`. Либо отдельная таблица `task_position_segments(sprint_id, task_id, segment_index, start_day, start_part, duration)`.
- **API**: в GET отдаём `segments` (если есть); в POST/batch принимаем опциональный `segments`; при отсутствии считаем один отрезок `[start_day, start_part, duration]`.
- **Фронт**:
  - Тип: `TaskPosition` расширить: `segments?: Array<{ startDay, startPart, duration }>`.
  - Если `segments` задан: рендерить несколько полос (`OccupancyPhaseBar` по одному на отрезок или один компонент с несколькими блоками), валидация и расчёт эффективной длительности — по `segments`.
  - Режим «разбить на отрезки»: по ПКМ по полосе → «Разбить на отрезки» → полоса превращается в сетку квадратов (например по частям дня); клик переключает отрезок вкл/выкл; «Сохранить» формирует массив включённых отрезков и уходит в API.

**Плюсы**: одна сущность на задачу, минимум изменений в ключах и в логике «одна фаза — одна полоса»; обратная совместимость при отсутствии `segments`.  
**Минусы**: миграция БД (новая колонка или таблица), обновление API и валидации.

### Вариант B: Несколько записей в `task_positions` на задачу (не использован)

- Менять PK на `(sprint_id, task_id, segment_index)` и хранить один отрезок на строку.
- **Минусы**: дублирование assignee/is_qa по отрезкам, агрегация по task_id везде (отображение, валидация, батчи), больше изменений в API и в коде. Не рекомендуется.

## Что править в коде (при варианте A)

1. **Типы** (`types/index.ts`): добавить в `TaskPosition` опциональное поле `segments?: Array<{ startDay: number; startPart: number; duration: number }>`.
2. **БД**: миграция — добавить `segments JSONB` (или таблицу отрезков) и при чтении/записи маппить в/из типа.
3. **API** (`app/api/sprints/.../positions/`, batch): при GET возвращать `segments`; при POST/batch принимать и сохранять `segments`; при отсутствии — считать одну позицию одним отрезком.
4. **occupancyValidation**: ввести `getPositionSegments(position): Array<{ startCell, endCell }>` (если есть `segments` — по ним, иначе один отрезок из start_day/start_part/duration). Все проверки (dev/QA, пересечения по assignee, недоступные дни) переписать в терминах «все отрезки всех позиций» (например, проверять пересечения между любыми двумя отрезками разных задач у одного assignee; для dev/QA — сравнивать конец последнего dev-отрезка с началом первого QA-отрезка по задаче).
5. **OccupancyPhaseBar** (или обёртка в OccupancyTaskRow):
   - Если у позиции есть `segments`: рендерить несколько полос (или одну «разорванную» визуально), каждая — drag/resize при желании; контекстное меню на каждой или общее «Разбить на отрезки» только для «целой» фазы.
   - Либо: одна полоса = габарит; по ПКМ «Разбить на отрезки» → режим редактирования сетки (квадраты вкл/выкл) → сохранение в `segments`.
6. **Оценка / доп часть**: эффективная длительность = сумма `duration` по всем отрезкам позиции; `extraDuration = max(0, effectiveDuration - estimatedDuration)`.
7. **Контекстное меню**: в обработчике контекстного меню по полосе фазы (уже вызывается `onContextMenu(e, task, false)`) добавить пункт «Разбить на отрезки», ведущий в режим сетки квадратов. Либо отдельное контекстное меню именно для фазы (если не хотите смешивать с общим меню задачи).

---

## Обратная совместимость

- Старые записи без `segments`: на фронте и в валидации считать одну позицию одним отрезком `[start_day, start_part, duration]`.
- Старые клиенты: при GET можно не отдавать `segments` или отдавать; при POST без `segments` — писать один отрезок из основных полей и очищать `segments` в БД (чтобы новые клиенты видели одну полосу).

## Итог

- **Фазы плана** сейчас — один непрерывный блок на задачу; контекстное меню по полосе уже есть (общее с задачей), можно добавить пункт «Разбить на отрезки».
- **Валидация и оценка** завязаны на один диапазон на позицию; для дробления нужно ввести «отрезки» и пересчитывать эффективную занятость и проверки по ним.
- Внедрить дробление можно без смены концепции «одна позиция на задачу», добавив опциональный массив `segments` (вариант A), с правками в БД, API, типах, валидации и в компоненте полосы/режиме редактирования сетки.
