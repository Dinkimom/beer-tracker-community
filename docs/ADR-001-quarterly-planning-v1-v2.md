# ADR-001: Квартальное планирование — две кодовые базы (v1 и v2)

**Статус:** принято  
**Дата:** 2026-04-04  
**Контекст:** finding F4 ([findings-features](../.spec-workflow/specs/check-architecture-problems/findings-features.md)).

## Контекст

В репозитории параллельно существуют:

| Путь | Назначение |
|------|------------|
| `features/quarterly-planning-v2/` | Актуальный UI и логика страницы квартального планирования (`QuarterlyPlanningV2Page`), точка входа из `MainPageClient`. |
| `features/quarterly-planning/` | Прежняя реализация (legacy): часть компонентов и утилит ещё переиспользуется. |

Дополнительно типы и хелперы v2 используются API (`app/api/quarterly-plans/v2/`), `lib/api/quarterly.ts`, спринт-планер (`useSprintPagePlanPhases`, `OccupancyPlannedInSprintBars`), feature-planner (`quarterGrouping`).

Без явных правил легко появляются хаотичные импорты и двусторонняя сцепка v1 ↔ v2.

## Решение

1. **Канон для продукта и новой разработки — v2** (`quarterly-planning-v2`). Новые экраны, хуки и бизнес-логика квартального планирования добавляются туда (или в `lib/`, если код общий для нескольких доменов).

2. **v1 — legacy.** Не считается целевым местом для новых фич. Допустимо временно тянуть из v1 только узкие, стабильные куски (UI/утилиты), пока они не перенесены в v2 или в общий слой.

3. **Направление зависимостей:** допустимо **v2 → v1** как краткосрочный компромисс (текущий пример: `QuarterSelector`, `getCurrentQuarter` в `QuarterlyPlanningV2Page.tsx`). Нежелательно добавлять **v1 → v2** (сейчас отсутствует — так и оставляем), чтобы не плотить циклы и не закреплять legacy как «хаб».

4. **Спринт и прочие фичи** могут зависеть от **v2** (типы, `quarterSprints`, хелперы планов в спринте). Зависимость **только от v1** для нового кода избегать; если нужен общий код — вынести в `lib/` или продублировать минимальный слой в v2.

## Последствия

- Онбординг: читать [QUARTERLY_PLANNING.md](./QUARTERLY_PLANNING.md) и этот ADR перед изменениями в квартальном планировании.
- Техдолг: по возможности переносить оставшиеся импорты v2→v1 в v2 или `lib/`, затем сокращать объём v1.
- Удаление v1 не запланировано в одном шаге: только после миграции потребителей и договорённости команды.

## Связанные пути

- API: `app/api/quarterly-plans/`, `app/api/quarterly-plans/v2/route.ts`
- Клиент: `lib/api/quarterly.ts`
- Типы данных: `types/quarterly.ts`
