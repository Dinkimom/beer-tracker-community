# Product steering — Beer Tracker (Sprint Manager)

## Назначение

Веб-приложение для **планирования спринтов в Яндекс Трекере**: визуальное распределение задач по дням, разработчикам и свимлейнам, бэклог, фичи, квартальное планирование, burndown и цели спринта.

## Пользователи и контекст

- Команды, которые уже ведут задачи в **Яндекс Трекере** и хотят планировать спринт на доске, синхронизируя изменения с трекером.
- Доступ к данным трекера через **OAuth-токен пользователя** (основной сценарий); опционально серверный fallback.

## Ценность (что должно «работать хорошо»)

1. **Планер спринта** — DnD, длительность, позиции, связи задач, комментарии на таймлайне, занятость.
2. **Бэклог** — отбор, фильтры, перенос задач в спринт.
3. **Планировщик фич** — фичи и задачи в их составе.
4. **Квартальное планирование** — эпики/стори/задачи, фазы, автопланирование с учётом зависимостей и загрузки.

## Вне фокуса продукта (для ясности при фичах)

- Замена Яндекс Трекера как системы учёта задач (приложение **надстраивается** над API).
- Хранение пользовательских OAuth-токенов на сервере (токен — в браузере, см. tech steering).

## Качество и риски

- Поломки планера после релизов (перетаскивание карточек, сохранение позиций, сбои из-за параллельных запросов к API) — критичны.
- Расхождение кэша React Query и UI после мутаций — избегать; использовать существующие ключи и патчи (см. ARCHITECTURE.md).
